# Phase 10: i18n (EN/KR) + README Split â€” Design

## Goal

Add full internationalization (English/Korean) to CineLog, covering both UI chrome and TMDB film content, without regressing the Core Web Vitals work from Phases 2/7/8/9. Separately, split the currently-inline-bilingual `README.md` into an English-only root README plus a standalone Korean README.

These two deliverables ship together as Phase 10 since the doc split is a natural companion to the i18n feature, but they touch entirely different parts of the repo and can be implemented/reviewed independently.

**Naming note:** the URL locale segment is the short slug `kr` (as in `/kr/films/42`), distinct from the standard codes used where external libraries require them â€” TMDB's `language` param uses `ko-KR`, and Clerk's localization export is `koKR`. All three refer to the same Korean locale; the difference is just each system's own naming convention. A single `LOCALE_TO_TMDB_LANGUAGE` map (`{ en: 'en-US', kr: 'ko-KR' }`) in `src/lib/tmdb.ts` is the one place that translates between them.

## Non-goals

- No per-account locale preference (no DB schema change). Locale is per-browser via cookie, same for anonymous and logged-in users.
- No languages beyond English and Korean.
- No translation of TMDB cast/crew names, only translatable fields TMDB already localizes (title, overview, tagline, genre names).

## 1. Routing restructure

All existing routes move down one level under a new `[locale]` dynamic segment:

```
src/app/
  layout.tsx                 # true root: <html>, fonts, ClerkProvider, SpeedInsights (unchanged)
  [locale]/
    layout.tsx                # NEW: validates locale param, wraps children in NextIntlClientProvider
    (browse)/
      layout.tsx
      page.tsx
      @modal/
        default.tsx
        (.)films/[id]/page.tsx
    films/[id]/
      page.tsx
      actions.ts
    search/page.tsx
    profile/page.tsx
    sign-in/[[...sign-in]]/page.tsx
    sign-up/[[...sign-up]]/page.tsx
```

This is a mechanical move, not a redesign of the routing itself: route groups, the `@modal` parallel slot, and the `(.)films` intercepting convention are all relative to their own directory level, so moving the whole tree together under `[locale]/` preserves their existing behavior unchanged.

`src/app/layout.tsx` keeps everything that must not vary by locale (font loading, Clerk provider, Speed Insights, the `<html>` tag itself â€” though `lang` now needs to read from the resolved `params.locale` instead of the hardcoded `"en"`).

`src/app/[locale]/layout.tsx` is new: it validates that `params.locale` is one of the supported locales (404s via `notFound()` otherwise, per next-intl convention), loads that locale's messages, and wraps `children` in `NextIntlClientProvider` so client components can call `useTranslations()`.

The stray empty `src/app/(browse)/search/` directory (no `page.tsx` in it â€” real search page is `src/app/search/page.tsx`) is left as-is; it isn't a live route and isn't part of this move.

## 2. Library & message organization

Add `next-intl` as a dependency. Messages are namespaced JSON files per locale, not one flat dictionary:

```
messages/
  en.json
  kr.json
```

Each file is grouped by feature namespace (`nav`, `film`, `search`, `profile`, `auth`), e.g.:

```json
{
  "nav": { "myList": "My List", "signIn": "Sign In" },
  "film": { "watched": "Watched", "rate": "Rate this film" },
  "search": { "placeholder": "Search films..." }
}
```

Server Components read messages via `getTranslations()` (next-intl's async server API); Client Components (e.g. `RatingWidget`, `WatchedButton`, `NavbarSearch`) use `useTranslations()` from the `NextIntlClientProvider` context set up in `[locale]/layout.tsx`.

## 3. Locale detection & persistence

`src/proxy.ts` â€” this Next.js version's `middleware.ts` equivalent â€” composes next-intl's routing middleware alongside the existing `clerkMiddleware`:

1. On first visit to `/`, next-intl's middleware inspects `Accept-Language` and redirects to `/en` or `/kr`.
2. The chosen locale is persisted in a `NEXT_LOCALE` cookie; subsequent visits skip detection and use the cookie.
3. `clerkMiddleware`'s existing public-route matcher (`/`, `/films/(.*)`, `/api/(.*)`, `/sign-in(.*)`, `/sign-up(.*)`) is updated to account for the locale prefix (e.g. `/(en|kr)/films/(.*)`), and `auth.protect()` logic is otherwise unchanged.

No account-level sync: a logged-in user who switches locale on one browser doesn't affect another browser/device. This matches the existing model where nothing about the user's UI preferences is persisted server-side today.

## 4. TMDB content localization

`src/lib/tmdb.ts` functions gain a `language` parameter (`'en-US' | 'ko-KR'`), threaded from the active locale down through the call sites in `src/app/[locale]/(browse)/page.tsx`, `src/app/[locale]/films/[id]/page.tsx`, and `src/app/[locale]/search/page.tsx`:

```ts
export async function getTrending(language: TTmdbLanguage): Promise<ITmdbMovie[]> {
  const res = await fetch(`${TMDB_BASE}/trending/movie/week?language=${language}`, {
    headers: getHeaders(),
    ...CACHE_1H,
  });
  ...
}
```

Because Next's fetch cache keys on the full request (URL + options), adding `language` to the query string automatically creates separate cache entries per locale under the existing `revalidate: 3600` policy â€” no changes needed to the caching strategy itself, and no risk of an English response being served from a Korean-tagged cache entry or vice versa.

`getMovieCredits` is unaffected (cast/crew names aren't localized).

## 5. UI locale switcher

A small client component (`LocaleSwitcher`) sits in `Navbar`, next to the auth slot. It reads the current pathname and active locale, and renders a link that swaps the locale segment while preserving the rest of the path (`/kr/films/42` â†’ `/en/films/42`, not a bounce to `/en`). Built on next-intl's locale-aware navigation helpers (`Link`, `usePathname`, `useRouter` from `next-intl/navigation`) rather than hand-rolled string replacement.

## 6. Clerk localization

When the active locale is `kr`, `@clerk/localizations`'s `koKR` is passed to `ClerkProvider`'s `localization` prop (falls back to Clerk's default English otherwise). This is flagged in code with a short comment noting that `@clerk/localizations` is an experimental package per Clerk's own docs â€” acceptable for a learning project, but a reason to double check auth screens after any future Clerk version bump.

## 7. Performance framing

This phase is framed as a case study in *locale-aware caching* rather than a departure from the project's Core Web Vitals focus:

- Locale-prefixed routes mean `/en/*` and `/kr/*` remain independently cacheable/staticaly-renderable â€” no shared-cache regressions from adding a second language.
- The locale switcher itself must not introduce layout shift (sized consistently with the existing `Navbar` slots, following the same fixed-size-fallback pattern already used for the auth slot).
- Checkpoint: re-run the Lighthouse/CWV audit from `docs/perf-baseline.md` against `/kr/*` routes; results should match the existing `/en/*` (previously unprefixed) baselines, not regress.

## Testing / checkpoint criteria

- `/en` and `/kr` both resolve correctly from `/`, with correct `Accept-Language`-based redirect on first visit and cookie persistence afterward.
- Switching locale via the `LocaleSwitcher` preserves the current route (film detail, search query, etc.).
- TMDB-sourced content (trending, now playing, film detail, search results) renders in the selected language; cast/crew names are unaffected.
- Sign-in/sign-up screens render in Korean when locale is `kr`.
- Lighthouse/CWV metrics on `/kr/*` match `/en/*` baselines from `docs/perf-baseline.md`.
- `README.md` contains no Korean text; `README.ko.md` is a complete Korean counterpart; both cross-link.
